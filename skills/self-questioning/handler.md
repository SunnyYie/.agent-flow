# Skill: Self-Questioning（任务后自我质询）

> **结构化反思**。在每个任务 VERIFY 之后、REFLECT 之前自动执行，主动发现流程缺口、未用工具、知识缺口，驱动持续优化。

## Trigger

- 每个任务 VERIFY 阶段完成后自动触发
- 位于 VERIFY 和 REFLECT 之间，是 REFLECT 的前置步骤
- 不可跳过 — 跳过自我质询的 REFLECT = 无效反思

## Procedure

### Step 1: 执行 10 项结构化自查

逐项检查，每项必须给出 **具体证据**，不可只写 PASS/FAIL：

#### 类别 1: 流程合规 (1-3)

**[1] 搜索先行合规**
- 问题：每个子任务执行前都搜索了 Skills/Wiki/Soul 吗？
- 证据要求：列出每个子任务及对应的搜索动作；哪些子任务跳过了搜索？

**[2] 复杂度流程合规**
- 问题：是否按照复杂度对应的流程执行？
- 证据要求：当前复杂度等级 + 每个流程步骤是否遵循；快速路径是否跳过了不该跳的？

**[3] 文档化合规**
- 问题：分析和计划是否写入了文档？
- 证据要求：哪些分析写入了文件，哪些是"脑中完成"的？

#### 类别 2: 知识利用 (4-6)

**[4] 未用工具识别**
- 问题：哪些可用工具我没用到？
- 检查清单：多Agent并行、WebSearch、find-docs Skill、gh search code、Agent(Explore)
- 证据要求：列出可用但未使用的工具 + 为什么没用

**[5] 已知坑命中检测**
- 问题：是否踩了已记录的坑？为什么已有知识没防止这个错误？
- 证据要求：踩了哪个坑 + 对应的 wiki/Soul 条目 + 为什么没生效（没搜到？搜到了没按做？）

**[6] Wiki 读取时机**
- 问题：Wiki 是执行前读的还是犯错后才读的？
- 证据要求：每个 wiki 引用的读取时机 + 是否是犯错后才想起搜的

#### 类别 3: 效率分析 (7-9)

**[7] 并行机会识别**
- 问题：哪些子任务可以并行但串行执行了？
- 证据要求：列出可并行的子任务对 + 为什么当时没并行

**[8] 不必要重试**
- 问题：是否有不必要的重试或回溯？更好的前期知识能否避免？
- 证据要求：重试次数 + 原因 + 哪些知识可以避免

**[9] 模型效率**
- 问题：模型选择是否合适？haiku 就够的用了 sonnet？
- 证据要求：哪些子任务用了什么模型 + 是否过配

#### 类别 4: 知识缺口 (10)

**[10] 事后知识识别**
- 问题：任务开始时我不知道什么会让任务更快/更好？
- 证据要求：列出事前不知道但事后发现关键的知识点

### Step 2: 评分与分级

```text
每项 PASS = 0 分, FAIL = 1 分
总分 0-10:
  0-2: 优秀 — 流程执行良好
  3-4: 合格 — 有改进空间
  5-7: 需改进 — 多个流程缺口
  8-10: 严重 — 流程几乎未遵循
```

### Step 3: 写入自查报告

将结果写入 `.agent-flow/state/self-questioning-report.md`：

```markdown
# Self-Questioning Report: {任务描述}
Date: {日期}
Complexity: {Simple|Medium|Complex}
Score: {0-10} ({优秀/合格/需改进/严重})

## 流程合规
- [1] 搜索先行: {PASS/FAIL} — {具体证据}
- [2] 复杂度流程: {PASS/FAIL} — {具体证据}
- [3] 文档化: {PASS/FAIL} — {具体证据}

## 知识利用
- [4] 未用工具: {具体列表或"无"}
- [5] 已知坑命中: {具体列表或"无"}
- [6] Wiki 读取时机: {具体证据}

## 效率分析
- [7] 并行机会: {具体列表或"无"}
- [8] 不必要重试: {次数和原因}
- [9] 模型效率: {观察}

## 知识缺口
- [10] 事前未知: {列表}

## 晋升建议 (Promotions)
- Soul.md 临界知识区: {新增条目或"无"}
- Wiki pitfalls: {新增条目或"无"}
- Wiki patterns: {新增条目或"无"}
- Skills: {新增或更新或"无"}
```

### Step 4: 触发后续动作

根据自查结果，触发对应机制：

| 条件 | 触发动作 |
|------|---------|
| [5] 踩坑 ≥2 次且 wiki confidence ≥0.9 | → 触发 `critical-knowledge` skill，注入 Soul.md 临界知识区 |
| [4] 发现未用工具 | → 添加到 config.yaml critical_tools 列表 |
| [7] 发现并行机会 | → 创建 wiki/patterns/workflow/ 条目 |
| [10] 发现知识缺口 | → 创建 wiki/pitfalls/ 条目 |
| [1] 搜索跳过 | → 记录为 pitfall，强化思维链执行 |

**所有晋升动作不在自查阶段执行**，只记录在"晋升建议"区域。自查完成后进入 REFLECT 阶段，REFLECT 阶段执行这些晋升。

### Step 5: 创建自我质询完成标记（Hook 强制要求）

写入 `.agent-flow/state/.self-questioning-done`：

```
timestamp={ISO8601}
task={任务描述}
score={0-10}
```

**此标记至关重要**：`self-questioning-enforce.py` hook 会在 REFLECT 阶段（写入 Soul.md 时）检查此标记。无标记 → REFLECT 被阻断。只有完成自查并创建标记后，才能继续执行 REFLECT 写入操作。

## Rules

- **不可跳过**：自我质询是 REFLECT 的必要前置，跳过 = 反思不完整
- **具体证据**：每项必须写出具体子任务名、工具名、文件名，不接受空泛的"做得不好"
- **诚实评估**：全部 PASS 的自查报告比有 FAIL 的更可疑 — 没有完美的执行
- **归档保留**：自查报告保留在 state/ 目录，用于后续 pitfall 频率统计
- **不执行晋升**：自查阶段只识别和记录，不直接修改 Soul.md/Wiki/Skills
