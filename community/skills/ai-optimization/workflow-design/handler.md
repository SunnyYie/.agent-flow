# Skill: workflow-design

> 合并自: workflow-design + workflow-architecture + dag-workflow-design + workflow-fault-tolerance
> 灵感来源: agency-agents Workflow Architect — 发现隐式流程、映射所有分支、定义接口契约、DAG编排、容错恢复

## Trigger
当需要设计、梳理或优化一个工作流/流程时触发。包括但不限于：
- 发现和梳理隐式工作流
- 设计多 Agent 工作流系统的整体架构
- 设计多步骤自动化流程，步骤间有依赖关系
- 设计条件路由、并行执行策略
- 设计长时间运行流程的容错与断点续跑

## Procedure

### Phase 1: Discovery — 发现阶段（必须先做）

1. 读取所有路由文件、Worker/Job 定义、数据库迁移、配置文件
2. 找出所有隐式工作流（存在但未文档化的流程）
3. 标记每个工作流的触发方式（用户操作/API调用/定时任务/事件）
4. 搜索 `.agent-flow/skills/` 和 Wiki 中是否有相关流程记录

### Phase 2: Design — 梳理工作流树与接口契约

#### 2.1 绘制工作流树

- 绘制 Happy Path（所有步骤成功）
- 为每一步标注失败模式：

| 失败模式 | 说明 | 示例 |
|----------|------|------|
| 输入校验失败 | 具体错误，用户看到什么 | 字段缺失、格式错误 |
| 超时失败 | 超时时间，超时后如何恢复 | API 响应超时 |
| 瞬时故障 | 可重试+退避 | 网络抖动/限流 |
| 永久故障 | 立即失败+清理 | 无效输入/配额超限 |
| 部分失败 | 前面步骤创建了什么需要清理 | 第7步失败 → 前6步的副作用 |
| 并发冲突 | 同一资源同时被修改 | 乐观锁冲突 |

#### 2.2 定义接口契约

每次系统间交接必须有：

```
交接: [From] → [To]
  负载: { 字段: 类型 }
  成功响应: { 字段: 类型 }
  失败响应: { 错误码, 错误信息, 是否可重试 }
  超时: X秒 — 视为失败
  失败恢复: [恢复动作]
```

#### 2.3 定义可观测状态

每个步骤必须回答：
- 用户看到什么？
- 运维看到什么？
- 数据库当前状态？
- 日志记录了什么？

#### 2.4 编写清理清单

所有资源创建步骤必须有对应清理动作：

| 资源 | 创建于步骤 | 清理方式 |
|------|-----------|---------|

#### 2.5 推导测试用例

每个分支 = 一个测试用例

### Phase 3: Architecture — 编排器-路由器-执行器三层架构

#### 3.1 三层职责划分

| 层 | 职责 | 关键能力 | 不应做的事 |
|----|------|----------|-----------|
| **编排器** (Orchestrator) | 消息接收、上下文管理、工作流选择 | 平台适配、会话管理、系统提示词构建 | 不执行具体工作流逻辑 |
| **路由器** (Router) | 意图识别、工作流/Agent 匹配 | 名称解析、复杂度分类、回退策略 | 不执行 AI 调用 |
| **执行器** (Executor) | 工作流/DAG 执行、AI 子进程调度 | 拓扑排序、并发执行、隔离管理 | 不做消息路由决策 |

**职责边界**：每层只做自己的事。路由器不做执行，执行器不做路由，编排器不做具体逻辑。

#### 3.2 编排器设计

编排器是系统的入口，负责：

1. **平台适配**：统一不同平台的消息格式
   ```
   Telegram 消息 → 统一消息格式
   Slack 消息   → 统一消息格式
   CLI 输入     → 统一消息格式
   GitHub Hook  → 统一消息格式
   ```

2. **上下文构建**：为 AI 路由构建系统提示词
   - 列出所有可用工作流及其描述
   - 注入当前项目信息和会话历史
   - 定义路由指令（何时使用哪个工作流）

3. **工作流选择**：将消息传递给 AI，AI 输出 `/invoke-workflow <name>` 指令

#### 3.3 路由策略设计

采用多级降级策略：

| 优先级 | 匹配方式 | 示例 | 说明 |
|--------|----------|------|------|
| 1 | 精确匹配 | `archon-idea-to-pr` → idea-to-pr | 用户明确指定 |
| 2 | 忽略大小写 | `IDEA-TO-PR` → idea-to-pr | 大小写容错 |
| 3 | 后缀匹配 | `to-pr` → idea-to-pr | 部分名称匹配 |
| 4 | 子串匹配 | "create pr from idea" → idea-to-pr | 关键词匹配描述 |

**回退策略**：所有级别都不匹配时 → 通用助手工作流

#### 3.4 执行器设计

1. **运行设置**
   - 创建运行记录（数据库/文件）
   - 解析隔离环境（worktree）
   - 加载配置（AI provider、模型、权限）
   - 检测可恢复的先前失败运行

2. **DAG 执行**
   - 拓扑排序节点
   - 并发执行独立节点
   - 管理节点间数据传递
   - 处理条件路由和循环

3. **生命周期管理**
   - 隔离环境与运行绑定
   - 运行结束后清理（或保留供调试）
   - 失败运行的恢复支持

#### 3.5 隔离策略

| 隔离维度 | 实现方式 | 说明 |
|----------|----------|------|
| 文件系统 | Git Worktree | 每个运行在独立 worktree 中 |
| 进程 | 独立 AI 子进程 | 每个节点可生成独立 AI 会话 |
| 状态 | 独立运行记录 | 每个运行有自己的状态追踪 |

### Phase 4: DAG — 有向无环图设计

#### 4.1 识别节点

列出流程中的所有执行步骤，每个步骤是一个节点。按类型分类：

| 节点类型 | 说明 | 示例 |
|----------|------|------|
| AI 节点 | 需要 LLM 决策的步骤 | 规划、实现、审查、分类 |
| 确定性节点 | Shell/脚本执行的步骤 | 测试、lint、部署、数据转换 |
| 控制节点 | 流程控制步骤 | 条件分支、人工审批、取消 |

**关键原则**：AI 节点用于判断性任务，确定性节点用于验证性任务。混用两者形成"AI 判断 + 确定性验证"的可靠模式。

#### 4.2 构建依赖图

明确节点间的依赖关系（`depends_on`）：

```
plan → implement → validate → review
                  ↘       ↗
              implement-fixes
```

**依赖图检查规则**：
- 每个节点只依赖其直接上游（传递依赖自动满足）
- 无依赖的节点可并行执行
- **必须无环**——有环是设计错误，需拆分为循环结构

#### 4.3 拓扑排序与执行分层

BFS 拓扑排序将节点分为执行层：

```
Layer 0: [plan]                          ← 无依赖
Layer 1: [implement]                     ← 依赖 plan
Layer 2: [validate]                      ← 依赖 implement
Layer 3: [code-review, error-review]     ← 依赖 validate，可并行
Layer 4: [synthesize]                    ← 依赖 Layer 3 所有节点
```

- 同层节点可并行执行（无依赖冲突）
- 不同层之间串行执行
- 并行节点之间不得有共享可变状态

#### 4.4 配置触发规则

为每个节点选择触发规则，决定"上游满足什么条件时本节点才执行"：

| 规则 | 含义 | 适用场景 |
|------|------|----------|
| `all_success` | 所有上游都成功才执行 | 默认规则，标准串行/并行流程 |
| `one_success` | 任一上游成功即可执行 | 备选方案，任一路径成功即继续 |
| `none_failed_min_one_success` | 无失败且至少一个成功 | 合并多个可选上游的结果 |
| `all_done` | 所有上游完成即执行（不管成败） | 清理节点、报告节点 |

**典型用法**：
- 标准流程：`all_success`（默认）
- 条件分支汇聚：`one_success`（任一分支成功即可）
- 并行审查合并：`none_failed_min_one_success`（无失败即可综合）
- 收尾/清理：`all_done`（始终执行）

#### 4.5 添加条件路由

使用 `when` 条件和结构化输出实现分支：

**上游节点**输出结构化数据：

```yaml
- id: classify
  prompt: "Classify this issue. Output JSON with 'category' field."
  output_format:
    type: object
    properties:
      category: { type: string }
```

**下游节点**用条件判断：

```yaml
- id: fix-bug
  depends_on: [classify]
  when: "$classify.output.category == 'bug'"

- id: implement-feature
  depends_on: [classify]
  when: "$classify.output.category == 'feature'"
```

**条件路由检查**：
- 所有可能路径都有终止节点（避免悬挂节点）
- 每个条件分支最终汇聚或终止
- `when` 条件的覆盖集必须完整（考虑 else 情况）

#### 4.6 验证端到端流水线

将以上步骤组合为完整流水线，验证：

1. **从起点到终点**：每个路径都能到达终止节点
2. **无死锁**：不存在循环等待
3. **无孤立节点**：每个节点都有上游或下游
4. **并行安全**：并行节点操作不同的文件/资源

### Phase 5: Fault Tolerance — 容错与断点续跑

#### 5.1 错误分类

将可能的错误分为两类，这是容错设计的基础：

| 分类 | 特征 | 示例 | 处理策略 |
|------|------|------|----------|
| **FATAL（致命）** | 重试不会改变结果 | 认证失败、权限不足、API 余额耗尽、配置错误、文件不存在 | 永不重试，立即终止并升级 |
| **TRANSIENT（暂态）** | 重试可能成功 | 网络超时、API 限流(429/503)、临时资源不可用、并发冲突 | 自动重试，指数退避 |

**分类决策**：

```
错误重试后是否会成功？
├── 不会（原因在外部不可变条件）→ FATAL
└── 可能（原因在临时状态）→ TRANSIENT
```

**常见 API 错误码分类**：

| 错误码 | 分类 | 理由 |
|--------|------|------|
| 401/403 | FATAL | 认证/权限问题，重试无意义 |
| 402 | FATAL | 余额不足，需人工处理 |
| 404 | FATAL | 资源不存在，重试无意义 |
| 429 | TRANSIENT | 限流，等待后可成功 |
| 500 | TRANSIENT | 服务器临时故障 |
| 502/503 | TRANSIENT | 网关/服务临时不可用 |
| 504 | TRANSIENT | 网关超时 |

#### 5.2 配置重试策略

仅对 TRANSIENT 错误重试：

```yaml
retry:
  max_attempts: 3         # 1-5 次
  delay_ms: 5000          # 初始延迟 1-60 秒
  on_error: transient     # 'transient'（默认）或 'all'
```

**指数退避**：每次重试等待时间翻倍

```
尝试1: 立即执行
尝试2: 等待 delay_ms (如 5s)
尝试3: 等待 delay_ms × 2 (如 10s)
尝试4: 等待 delay_ms × 4 (如 20s)
```

**`on_error` 策略**：
- `transient`（默认）：仅重试 TRANSIENT 错误，FATAL 直接升级
- `all`：重试所有错误（谨慎使用，仅当不确定错误分类时）

#### 5.3 实现断点续跑

记录每个步骤的完成状态，恢复时自动跳过已完成步骤：

**状态记录格式**：

```json
{
  "workflow_id": "run-001",
  "nodes": {
    "plan": { "status": "completed", "output_ref": "/artifacts/plan.md" },
    "implement": { "status": "completed", "output_ref": "/artifacts/changes.md" },
    "validate": { "status": "failed", "error": "TRANSIENT: timeout", "attempts": 2 },
    "review": { "status": "pending" },
    "finalize": { "status": "pending" }
  }
}
```

**恢复逻辑**：

| 节点状态 | 恢复时处理 |
|----------|-----------|
| `completed` | 跳过，从 output_ref 加载输出 |
| `failed` (TRANSIENT) | 重新执行 |
| `failed` (FATAL) | 不重试，升级人工 |
| `pending` | 正常执行 |

**幂等性要求**：已完成步骤重跑不得产生副作用（如重复创建 PR、重复提交代码）。

#### 5.4 人工介入点

| 场景 | 处理方式 |
|------|----------|
| 超过最大重试次数 | 暂停，报告问题详情，等待人工决策 |
| FATAL 错误 | 立即暂停，不自动重试 |
| 人工介入后可选 | 重试当前步骤 / 跳过当前步骤 / 终止整个流程 |

**人工介入报告格式**：

```
[FAULT-ALERT] 工作流 {workflow_id} 需要人工介入
## 失败节点
- 节点: {node_id}
- 错误类型: FATAL / TRANSIENT (exceeded max_attempts)
- 错误详情: {error_message}
- 已重试: {attempts} 次
## 已完成节点
- plan: completed
- implement: completed
## 可选操作
1. 重试当前节点
2. 跳过当前节点（风险：下游可能缺少该节点输出）
3. 终止工作流
```

## Rules

**通用规则**：
- 不只为 Happy Path 设计，必须覆盖所有失败分支
- 不跳过可观测状态定义
- 不留下未定义的接口交接
- 不将无关工作流捆绑在同一文档中
- 每个超时假设必须标记（可能产生竞态条件）

**架构规则**：
- 三层职责不得混淆——路由器不做执行，执行器不做路由，编排器不做具体逻辑
- 路由失败必须有明确回退（如通用助手），不得静默丢弃消息
- 隔离环境的生命周期必须与工作流运行绑定——运行结束即清理
- 编排器的系统提示词必须列出所有可用工作流，否则 AI 无法正确路由
- 新增平台适配器只需实现统一消息接口，无需修改编排器逻辑

**DAG 规则**：
- DAG 必须无环——有环是设计错误，应拆分为循环结构
- 每个节点必须有明确的输入来源和输出去向
- 条件路由的所有分支必须最终汇聚或终止
- 并行节点之间不得有共享可变状态（避免竞态条件）
- 人工审批节点必须有超时和拒绝处理路径
- AI 节点后必须跟确定性验证节点（AI 判断 + 确定性验证）

**容错规则**：
- FATAL 错误绝不自动重试——重试会浪费资源且不会成功
- 重试间隔必须指数退避，避免加重系统负载（尤其是限流场景）
- 断点续跑必须幂等——已完成步骤重跑不得产生副作用
- 每次重试必须记录日志（错误类型、重试次数、等待时间）
- 人工介入后选择"跳过"时，必须评估下游节点是否受影响
- `on_error: all` 仅在不确定错误分类时使用，不应作为默认策略
