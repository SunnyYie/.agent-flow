# Pattern: 测试矩阵5维检查

> 建立测试验收矩阵时必须执行5维检查，防止原始问题覆盖出现盲区。
> 来源：Maiya V1.5 步骤18双Agent验收实践。

## 问题

项目文档经过多次重组后，原始问题（P-x/R-x）容易在验收矩阵中"消失"——表现为：
- 只标注"间接覆盖"而无量化通过标准
- 验收项无法反查到原始问题编号
- 自动化测试写了但未映射到具体验收项
- 边界场景（冲突/过期/重复）被遗漏
- 单视角验收盲区

## 5维检查清单

### 1. 覆盖度

**每个原始问题必须有直接验收项，不允许只有"间接覆盖"。**

```
检查方法：
├── 遍历所有原始问题编号（P-01~P-10, R-01~R-06等）
├── 每个编号在验收矩阵中至少对应1个直接验收项
├── 验收项含量化通过标准（如"P95 < X秒"，而非"优化后变快"）
└── 不允许"间接覆盖"表述（如"通过XX功能间接解决"）
```

**反例**：P-09标注为"P2项，间接覆盖（失败恢复减少重试耗时）"→ 无量化标准，无法证明问题被解决

**正例**：P-09直接验收项"页面级骨架更新P95耗时相对基线下降≥30%"

### 2. 可追溯

**验收项→原问题→冻结约束三者必须可双向反查。**

```
检查方法：
├── 正向查：每个验收项标注"覆盖原问题编号"
├── 反向查：每个原问题在覆盖矩阵中有去向和验收凭证
├── 纵向查：验收项与冻结约束编号对齐
└── 不允许存在"只写方案、不写问题来源"的验收项
```

### 3. 自动化映射

**每个 automated_test 验收项必须映射到具体测试文件和用例名。**

```
检查方法：
├── 验收矩阵增加"自动化测试映射"列
├── 格式：`文件名 :: 用例名`（如 `step18-test-matrix.test.ts :: "版本冲突检测"`）
├── 映射目标文件必须存在（Glob验证）
└── manual_test 和 integration_test 不要求映射，但需标注验证方式
```

### 4. 边界场景

**测试矩阵必须覆盖冲突优先级、幂等保护、过期处理、重复操作。**

```
每个功能模块至少3个边界测试用例：
├── 冲突：版本冲突/权限冲突/并发冲突
├── 过期：预览过期/快照过期/缓存过期
└── 重复：重复确认应用/重复撤销/幂等重试

遗漏信号：
├── 只测了主流程的 happy path
├── 状态机守卫只验证合法迁移，未验证非法迁移
└── 只检查"返回了正确结果"，未检查"错误时返回了正确错误码"
```

### 5. 双验收

**复杂度≥Medium的验收必须双Agent独立审查，交叉比对发现盲区。**

```
双验收执行方式：
├── Verifier Agent：独立审查所有产出物，输出结构化审查报告
├── Main Agent：执行原始问题覆盖检查，逐项核对
├── 交叉比对：两者发现的遗漏项取并集
└── 修复后重新验证：遗漏项修复后至少单Agent复查

典型发现（来自实践）：
├── P-09直接验收项缺失（单视角容易忽略P2问题）
├── 联调脚本场景结构不完整（审查者更关注一致性）
├── require()不兼容ESM（单视角容易忽略运行时兼容性）
└── 映射表产出确认（审查者关注"写了但没有产出"的问题）
```

## 执行时机

- 步骤验收阶段（VERIFY前）
- 文档重组后（防止重组丢问题）
- 发布前人工验收清单检查

## 与其他模式的关系

- **三段式联调脚本**：5维检查中的"自动化映射"维度依赖联调脚本的完整结构
- **orchestrator-workers**：5维检查中的"双验收"维度使用多Agent并行验证
- **testing-strategies**：5维检查是测试策略的质量门控补充
