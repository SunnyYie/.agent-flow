# Shared Skill: AI 上下文管理策略

## Trigger — 何时使用

- 设计多 Agent 或多步骤 AI 流程时，需要决定 Agent 间如何传递信息
- AI 迭代循环中需要决定每轮是否共享上下文（参考 `iterative_ai_refinement.md`）
- 长时间运行的 AI 任务面临上下文窗口溢出风险
- Agent 子进程之间需要传递大量结构化数据

- `.dev-workflow/shared/references/archon.md` — Archon 上下文管理设计

## Procedure — 执行步骤

### Step 1: 评估上下文需求

设计多步骤流程时，先评估每对上下游步骤间的信息传递需求：

| 评估维度 | 问题 | 影响 |
|----------|------|------|
| **信息量** | 需要传递多少数据？ | 少量变量 → 变量替换；大量内容 → 文件传递 |
| **时效性** | 需要实时状态还是快照即可？ | 实时 → 共享会话；快照 → 文件/状态对象 |
| **独立性** | 下游是否依赖上游的推理过程？ | 仅需结果 → 文件/变量；需推理链 → 共享会话 |
| **安全性** | 是否包含敏感信息？ | 包含 → 必须脱敏或特殊通道 |

### Step 2: 选择传递机制

| 机制 | 适用场景 | 优点 | 缺点 | 示例 |
|------|----------|------|------|------|
| **会话内继承** (shared context) | 步骤间需要推理链连续性 | 保留完整上下文，AI 理解修改历史 | 上下文窗口有限，长流程溢出 | 同一会话内连续修复 |
| **产物文件** ($ARTIFACTS_DIR) | 步骤间传递大量结构化数据 | 不受窗口限制，可持久化 | 丢失推理上下文 | plan.md → implement 读取 |
| **变量替换** ($nodeId.output) | 步骤间传递少量关键信息 | 轻量精确，运行时解析 | 仅适合小数据，不适合长文本 | 分类结果传递到下游 |
| **状态对象** (dict/JSON) | 编程式工作流 (LangGraph 等) | 类型安全、可校验、可序列化 | 需要预定义 schema | WorkflowState TypedDict |

**选择决策**：

```
信息量多大？
├── 少量（< 1KB 关键信息）→ 变量替换
├── 中等（1KB-100KB 结构化数据）→ 状态对象 或 产物文件
└── 大量（> 100KB 文件内容）→ 产物文件（路径引用）

下游是否需要上游推理过程？
├── 是 → 会话内继承（但注意窗口限制）
└── 否 → 产物文件 或 状态对象
```

### Step 3: 防止上下文溢出

上下文窗口是有限资源，必须主动管理：

| 阈值 | 策略 |
|------|------|
| ≤ 50% 窗口 | 正常使用，无特殊处理 |
| 50%-70% 窗口 | 压缩历史信息，保留关键摘要 |
| > 70% 窗口 | 切换为 fresh context + 摘要传递 |

**具体防护措施**：

1. **大文件通过路径引用而非内容内联**
   ```
   # 差：内联整个文件内容
   prompt: "Here is the code: {entire_file_content}"

   # 好：引用文件路径
   prompt: "Read the plan at $ARTIFACTS_DIR/plan.md"
   ```

2. **长对话切换为 fresh context + 摘要**
   ```
   # 在上下文使用超过 70% 时
   # 1. 生成当前状态摘要
   # 2. 新会话中注入摘要 + 剩余任务
   # 3. 通过产物文件保留详细历史
   ```

3. **批量信息分页传递**
   ```
   # 差：一次性传入 100 个任务
   # 好：分批传入，每批 10 个，通过进度文件跟踪
   ```

### Step 4: 上下文注入标准化

按三个部分组织上下文注入：

```
┌─────────────────────────────────────┐
│ 固定部分（系统提示词）               │
│ - 角色定义、安全规则、行为约束        │
│ - 每次会话相同，不随任务变化          │
│ → 系统提示词层                       │
├─────────────────────────────────────┤
│ 动态部分（任务数据）                  │
│ - 当前任务描述、输入数据              │
│ - 通过变量或文件注入                  │
│ → 变量替换 / 产物文件                 │
├─────────────────────────────────────┤
│ 历史部分（前序产出）                  │
│ - 上游节点的输出、之前的决策          │
│ - 按需摘要后注入                     │
│ → $nodeId.output / 摘要文件          │
└─────────────────────────────────────┘
```

**注入优先级**：固定部分 > 动态部分 > 历史部分。上下文不足时优先裁剪历史部分。

### Step 5: 并行上下文隔离

多个并行 Agent 必须使用独立上下文：

```
                    ┌→ Agent A (fresh context, 独立会话)
upstream output ────├→ Agent B (fresh context, 独立会话)
                    └→ Agent C (fresh context, 独立会话)
```

- 每个并行 Agent 使用 **fresh context**，避免观点交叉影响
- 并行 Agent 通过共享的只读文件获取输入（如 $ARTIFACTS_DIR）
- 并行 Agent 的输出各自独立，由下游合并节点统一处理

## 与 LangGraph 状态传递的关系

本项目的 agent-workflow 使用 LangGraph 状态对象传递上下文：

| LangGraph 机制 | 本技能对应 | 说明 |
|----------------|-----------|------|
| `Annotated[dict, merge_dicts]` | 状态对象 | 并行节点通过 reducer 合并 |
| `add_conditional_edges` | 变量替换 | 条件路由基于状态字段值 |
| `interrupt_after` | 交互循环 | 暂停等待人工输入 |
| `thread_id` + checkpoint | 断点续跑 | 会话持久化与恢复 |

LangGraph 的状态对象是本技能"状态对象"机制的具体实现。

## Rules — 规则约束

- 敏感信息（密钥、token）不得通过上下文传递机制传递——使用环境变量或安全的 secret store
- 上下文注入必须有明确边界——不注入与当前任务无关的信息
- fresh context 模式下，必须确保所有必要信息通过文件/变量传递（不能依赖会话记忆）
- 并行 Agent 必须使用 fresh context，避免上下文交叉污染
- 大文件必须通过路径引用传递，不得内联到上下文中
- 参考现有技能：`git_operations.md` 的 worktree 隔离确保并行上下文不冲突
