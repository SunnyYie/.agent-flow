# 需求文档中前后端功能分类

## 问题描述

需求规格说明书通常只描述"做什么"，不区分功能应该由前端还是后端实现。这导致：
- 前后端开发边界不清，职责重叠或遗漏
- 任务分配时无法准确拆分前端工单和后端工单
- 验收时缺少端侧维度的检查

## 解决方案

在需求规格说明书中增加"功能端侧归属"维度，对每个功能点标注前端/后端/前后端归属。

### 核心判断规则

| 判断条件 | 归属端 | 示例 |
|----------|--------|------|
| 用户直接可见的 UI 展示和交互 | 前端 | 按钮点击、页面跳转、表单填写 |
| 客户端数据采集与上报 | 前端 | 埋点上报、曝光/点击统计 |
| 客户端本地状态管理 | 前端 | 离线缓存、本地偏好设置 |
| 纯服务端业务逻辑 | 后端 | 定时任务、消息队列、数据清洗 |
| 第三方服务集成（服务端调用） | 后端 | AI模型调用、OCR服务、推送 |
| 数据持久化与查询 | 后端 | 数据库CRUD、缓存管理 |
| 后台自动化操作 | 后端 | 自动发帖、自动回复、定时调度 |

### 模糊场景判断

| 场景 | 判断 | 原因 |
|------|------|------|
| 数据埋点 | **前端** | 客户端采集用户行为并上报 |
| 消息推送展示 | 前端(展示) + 后端(推送) | 后端触发，前端展示 |
| 表单校验 | 后端(主) + 前端(辅) | 后端安全校验为主，前端体验为辅 |
| 文件上传 | 前端(选文件) + 后端(存储) | 前端选择，后端处理 |

### 输出格式

在需求规格说明书中增加：

1. **功能需求正文标注**：每个功能小节标题后追加 `[前端]` / `[后端]` / `[前后端]`
2. **功能端侧归属汇总表**：

```markdown
## 功能端侧归属

| 功能点 | 归属端 | 判断依据 |
|--------|--------|----------|
| {功能1} | 后端 | 服务端业务逻辑，用户不可见 |
| {功能2} | 前端 | 客户端数据采集与上报 |
| {功能3} | 后端(主) + 前端(辅) | 后端执行+前端展示 |
```

## 实际案例：AI辅助简历诊断

**需求来源**：飞书 Wiki `https://maimai.feishu.cn/wiki/NCY6wK30OivNSSkfaKVckE4QnN5`

**分类结果**：

| 功能点 | 归属端 | 判断依据 |
|--------|--------|----------|
| 触发检测（监控新帖子，识别关键词） | 后端 | 服务端定时扫描，用户不可见 |
| 附件检查（帖子含图片附件） | 后端 | 服务端逻辑判断 |
| OCR 识别（识别图片文字） | 后端 | 第三方服务调用，服务端执行 |
| 求职意向识别 | 后端 | 服务端文本处理逻辑 |
| Agent 诊断（AI 生成诊断建议） | 后端 | AI 模型调用，服务端执行 |
| 自动回帖（AI 助手评论） | 后端 | 服务端自动化操作 |
| 频率限制（同一帖子只回复一次） | 后端 | 服务端业务规则 |
| 审核规则（AI 评论走生态审核） | 后端 | 服务端审核流程 |
| 字数限制截取 | 后端 | 服务端文本处理 |
| 数据埋点（AI 评论曝光/点击统计） | **前端** | 客户端采集用户行为并上报 |

**关键洞察**：这个需求文档中，**只有数据埋点属于前端功能**，其余全部是后端实现。这是一个典型的"后端驱动 + 前端埋点"模式——前端几乎不需要开发新功能，只需在已有评论组件上增加埋点事件。
